## 2) API 스펙(최소)

### Endpoint
- `POST /api/recommend`

### Request(JSON)
```json
{
  "preset": "A",
  "games": 5,
  "seed": 42
}
````

* preset:

  * A = 보수적(필터 강함)
  * B = 공격적(필터 약함)
* games: 5(기본), 10(옵션)
* seed: optional (없으면 서버에서 생성)

### Response(JSON)

```json
{
  "meta": {
    "preset": "A",
    "percentile": 12
  },
  "recommendations": [
    {
      "numbers": [1,2,3,4,5,6],
      "score": 0.123,
      "tags": ["hmm_hot"],
      "reasons": ["AC>=7", "sum 100-175", "zone<=3"]
    }
  ]
}
```

* percentile: (있으면) “이 설정이 과거 기준 어느 정도인지” 한 숫자
* reasons: 프론트에서 **근거로 표시** (없으면 표시 생략 가능)
* tags: 있으면 배지로 표시

> **중요:** 백엔드 결과 형식이 다르면 `adapter` 계층에서 위 형태로 변환만 하면 됨.

## 3) 프론트 1페이지 요구사항

### 섹션 1: 헤더(정체성)

* 제목: `확률 실험 기반 선택 보조 도구`
* 서브: `당첨을 보장하지 않습니다. 무근거 선택을 제거합니다.`
* 배지: `이번 설정: 과거 기준 상위 X%` (X 있으면 표시)

### 섹션 2: 입력(최소)

* Preset 버튼 2개: A/B
* games: 5 / 10 토글
* seed: 고급옵션(토글) 안에만

### 섹션 3: 결과(핵심)

* 추천 테이블

  * 번호 6개
  * score
  * tags(있으면)
  * reasons(있으면)
* 버튼: `다시 추천`

### 섹션 4: 결제 CTA

* 문구: `매주 자동 리포트를 받고 싶다면`
* 버튼: `월 9,900원` → `PAYMENT_URL`로 이동

### 섹션 5: 면책(짧게)

* 통계/실험 목적
* 당첨 보장 없음
* 구매 판단은 사용자 책임

---

## 4) 구현 선택지(오늘 완성 기준)

### 선택지 A: FastAPI + Next.js (권장)

* 장점: 추후 확장 쉬움, UI 보기 좋음
* 오늘 기준 최소:

  * FastAPI: endpoint 1개
  * Next: 단일 페이지 + fetch

### 선택지 B: Streamlit 단독(가장 빠름)

* 장점: 1~2시간 컷 가능
* 단점: “상품 느낌”은 약함
* 결제 버튼 링크는 가능
